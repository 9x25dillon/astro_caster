# MOBILE_ROADMAP.md — Astra Mobile Counterpart: Technical Direction & Execution Plan

_Status: **Living** — owner-reviewed roadmap. Created 2026-07-04 on
`claude/resonarium-biosentinel-integration-6gzi6a` (PR #20)._

---

## 0. Executive summary

Astra already runs on a phone — but as a *hosted web app inside Termux*, which
is a developer posture, not a product. The mobile counterpart is not "a mobile
app that calls our server." Astra's identity is **deterministic, offline-first,
privacy-absolute** (birth data never leaves the device). The only mobile
architecture that *keeps the claims true* is one where the deterministic core
runs **on the device itself**.

The strategy is therefore a three-horizon **compute gradient** — each horizon
ships a usable product while funding the next:

| Horizon | Ship vehicle | Where compute lives | Timeframe |
|---|---|---|---|
| **H1 — Installable now** | Hardened PWA (exists, needs finishing) | Backend on LAN/hosted; PWA shell offline | weeks |
| **H2 — Store-distributable** | Capacitor wrapper around the same build | Thin hosted API for chart math + AI relay | 1–2 quarters |
| **H3 — True counterpart** | Same shell + **ASTRA-CORE on device** | Everything deterministic on-device; network only for opt-in AI | 2–4 quarters |

The single load-bearing initiative is **ASTRA-CORE** (§3): extracting the
deterministic engines into a TypeScript package proven equivalent to the
Python backend by golden-vector parity CI — the exact methodology already
validated in this repo by `resonarium/natal_seed.{py,js}` (bit-identical
seeds, PRNG streams, and modulation across Python and JS, enforced by
`resonarium/tests/test_biosentinel.py::TestBrowserParity`).

## 1. What exists today (verified, not assumed)

- **Frontend is already a PWA**: `vite-plugin-pwa` with `autoUpdate`,
  standalone display, SVG icon, ~562 KiB precache. It installs to a home
  screen today; it does not yet *work* without the backend.
- **The engines are small.** The entire deterministic core is ~2,000 lines:
  `ephemeris.py` (387), `forecast.py` (493), `tarot.py` (700) + data,
  `astrology.py`, `synastry.py`, `advanced.py`, `predictive.py`. Numerology
  and the print/sigil pipeline are *already* TypeScript on the client.
- **PDF generation is already client-side** (`lib/printReport.ts`) — verified
  working through Android's print dialog this session. Nothing to port.
- **Entitlements are HMAC-signed server-verified tokens** — this is the one
  subsystem that *cannot* move on-device as-is (§5).
- **Mobile unlock affordance** (`?entitlement=<token>`) and the Termux
  bring-up path exist and are documented in `TESTING.md`.
- **Resonarium** is already fully client-side and offline by construction.

## 2. The architectural decision (H1→H2→H3), and why not the alternatives

**Chosen: PWA-first, wrapped by Capacitor, converging on on-device compute.**
One codebase, the existing React/TS/D3 investment intact, the store presence
when we want it, and the offline-first claim made literal.

Rejected alternatives, for the record:

- **React Native / Flutter rewrite** — discards a mature, tested D3-SVG wheel
  and ~7k lines of working UI for zero user-visible gain. The wheel *is* the
  product surface; SVG in a WebView is its native habitat.
- **Backend-in-app (Python-on-Android à la Chaquopy/Termux)** — what we do
  today for development. Unshippable: py-native builds per ABI, 100 MB+
  payloads, store review friction, and pyswisseph compilation fragility
  (observed first-hand on Termux this week).
- **Hosted-backend-only mobile client** — simplest, but it *inverts the
  privacy claim*: every birth chart would transit a server. Acceptable only
  as the H1/H2 interim, never as the end state.

## 3. ASTRA-CORE — the load-bearing initiative

**Goal:** `@astra/core` — a dependency-free TypeScript package computing
charts, transits, aspects, patterns, dignities, tarot mappings, forecast
events, synastry/composite/Davison, progressions, returns, harmonics,
midpoints, and fixed stars — **provably equivalent** to the Python backend.

**The normally-unidentified solution — parity as a product, not a QA step:**
we do not "port and hope." We extend the pattern that already ships in this
repo: every engine function gets golden vectors generated by the Python
implementation (`backend/tools/gen_parity_vectors.py` → `parity/*.json`, committed),
and the TS implementation must reproduce them within stated tolerances in CI
— on every commit, forever. Divergence is a build failure, not a bug report.
This converts the scariest multi-year risk (silent astronomical drift between
implementations) into a mechanical gate.

**Ephemeris strategy (the hard 20%):**
- Primary: **`astronomy-engine`** (MIT, pure TS, ~119 KB, Moshier-class
  accuracy ±1 arcmin for our bodies) — matches the backend's own Moshier
  fallback posture, which is what production runs today (`ephemeris:
  "moshier"` in `/api/health`).
- Escalation path if parity tolerances demand it: **Swiss Ephemeris compiled
  to WASM** behind the same interface (adds ~600 KB; decision deferred until
  vectors say we need it).
- Tolerance contract per body class, versioned in the parity manifest:
  longitudes ±0.01°, house cusps ±0.02°, aspect orbs ±0.01°, event dates
  exact-day. Tarot/forecast/numerology: exact equality (they're arithmetic).

**Sequencing inside the initiative** (each step independently shippable):
1. ✅ **v0.1 DONE 2026-07-05** — `@astra/core/chart` (`packages/astra-core/`):
   positions, houses (Placidus from scratch), angles, aspects, dignities,
   elements/modalities, patterns — reproduces `parity/natal-chart.json` within
   the cross-engine tolerance (worst longitude Δ ~0.003°, cusps ~0.001°) using
   `astronomy-engine`, an *independent* Moshier implementation. CI job
   `astra-core` gates every commit. **Known gap:** Node/Chiron/Lilith need an
   ephemeris source astronomy-engine lacks — comparison restricts to the
   supported body set; closing it is the WASM-Swiss escalation below.
   *Unblocks offline chart casting.*
2. ✅ **v0.2 DONE 2026-07-05** — `@astra/core/tarot`: natal signature + weighted
   spread draws, **bit-exact** to the backend. The "mechanical port" framing
   undersold it — the draw seeds CPython's `random.Random`, so it required a
   Python-compatible **MT19937** (`mt19937.ts`: `init_by_array` seeding from the
   256-bit sha256, `genrand_res53`), proven against `parity/mt19937.json`
   independently of tarot. Draws proven against `parity/tarot-draw.json` (exact,
   not tolerance). Two traps caught: the backend joins seed parts with an
   invisible U+0001 separator (`"\x01".join`), and weights are Python
   round-half-even before feeding the RNG. Deck/weight data generated from the
   Python source (DRY). Prose/lessons/learning-path deferred (static lookups).
3. ✅ **v0.3 DONE 2026-07-05** — `@astra/core/forecast` (`forecast.ts`):
   day-by-day transit scan — stations (speed-sign bisection), transit-to-transit
   and transit-to-natal exactness (Moon at 6h resolution), hysteresis + 10-day
   dedup — a faithful port of `forecast.py`. Reproduces the backend's ~120-event
   60-day forecast for both reference natals against `parity/forecast.json`,
   matched by event identity within a ±1-day / 0.2° cross-engine window (near-
   midnight stations and flat-minimum aspects can shift a day between
   astronomy-engine and pyswisseph). Sun–Pluto transits (v0.1 body set). Still
   synchronous; the Web-Worker chunking/cancellation is a client-integration
   concern for when it wires into the UI.
4. `@astra/core/relational` + `predictive` + `advanced` — long tail, gated by
   usage telemetry (which modals people actually open on mobile).
5. Backend adopts the same vectors as *its* regression suite — the two stacks
   are now pinned to each other symmetrically.

## 4. Subsystem plans

### 4.1 Shell & distribution
- **H1:** finish the PWA — offline app-shell (works with backend absent:
  cached last chart, full tarot/numerology/sigil/resonarium, queued AI asks),
  install prompts, iOS safe-areas, `share_target` for chart JSON import.
  Self-host the two webfonts (removes the app's only external request —
  flagged in this session's audit).
  - ✅ **2026-07-05** — on-device **chart casting** landed: `@astra/core`
    (chart + forecast engines) wired into the frontend via a browser-safe
    entry; `generate()` degrades API → same-birth cache → on-device compute.
    astronomy-engine bundled (no external request). Reduced body set on the
    local path (no Node/Chiron/Lilith) until the WASM-Swiss escalation.
    Remaining H1: on-device tarot (needs isomorphic sha256), queued AI asks,
    install prompts, iOS safe-areas, `share_target`.
- **H2:** Capacitor wrapper: share-sheet PDF export (replacing print-dialog
  UX), haptics on wheel selection, keep-awake during Resonarium sessions,
  file-system access for state import/export. One repo, `mobile/` directory,
  same `dist/`.
- Touch pass on the wheel: 44 px hit targets (hit-line widths are already
  generous), pinch-zoom on the wheel, long-press = hover-popover equivalence.

### 4.2 Entitlements & payments (the policy trap most plans miss)
- **Verification must become asymmetric.** HMAC tokens can't be verified
  on-device without shipping the secret. Migrate to **Ed25519 signatures**:
  server signs, device verifies with the embedded public key. Same claims
  payload, same expiry semantics; `entitlements.py` gains a keypair alongside
  the HMAC path (dual-issue during migration; HMAC retired at H3).
- **Store policy risk (systemic, multi-year):** crypto-donation unlocks
  violate Apple/Google IAP rules *for store builds*. Strategy: PWA channel
  keeps the open-paywall/crypto rail untouched; store builds ship either
  (a) reader-mode (all free features, tier import via file/QR — the Kindle
  pattern), or (b) native IAP mapped to the same signed tokens. Decide per
  store at H2 entry; the token layer is deliberately agnostic.

### 4.3 AI layer
- Keys never touch the device. A thin **relay** (the existing FastAPI app
  deployed small — Fly/Railway class) does tier routing (Haiku/Sonnet/Opus),
  rate limiting (`ratelimit.py` exists), and returns SSE streams the client
  already parses. Offline: the deterministic fallback ships *in* the app —
  the client-side mirror of `ai.py`'s offline synthesizer, fed by ASTRA-CORE.
- Oracle/personal-report generation stays relay-side (model quality is the
  paid claim); compiled markdown renders and prints on-device as today.

### 4.4 Resonarium
Already the most mobile-ready subsystem (client-only, no network, CSP-locked).
Mobile work: audio-session handling (interruption/resume on calls),
keep-awake, background-audio decision (default **no** — the visual field is
half the instrument; document it), and the existing photosensitivity gate
carries over unchanged.

### 4.5 Observability & quality gates
- Extend the session-built Playwright suites into `frontend/e2e/` with mobile
  viewports/emulation as the PR gate (they currently live only in scratchpad).
- Device lab minimum: 1 low-end Android (WebView fragmentation canary),
  1 iOS Safari (PWA restrictions canary). WebKit-without-push/badging is a
  known iOS constraint — features degrade, never break.
- Telemetry stays consent-based and local-first (existing `telemetry.py`
  posture); mobile adds only feature-usage counters needed for §3 step 4.

## 5. Systemic multi-year risks & their standing mitigations

| Risk | Horizon | Mitigation (standing, not one-off) |
|---|---|---|
| Silent Python↔TS astronomical drift | permanent | Golden-vector parity CI (§3); tolerance manifest versioned; drift = red build |
| App-store policy vs. crypto paywall | H2+ | Dual-channel distribution (§4.2); token layer payment-agnostic by design |
| WebView/Safari fragmentation | permanent | Two-canary device lab; feature-detect + degrade; no API used without a fallback already written |
| Ephemeris dependency abandonment | H3+ | Interface-pinned ephemeris (§3); WASM Swiss Ephemeris as pre-scoped escalation |
| Key/secret management as surface grows | H2+ | Ed25519 verify-only on device; signing keys never leave server; rotation runbook in `entitlements.py` docs |
| Termux path rot (current dev channel) | H1 | H1 PWA makes Termux optional; keep `TESTING.md` §2 mobile path green in CI smoke |
| Scope gravity (porting the long tail) | H3 | Usage-gated sequencing (§3 step 4); "no port without a vector file" rule |

## 6. Phase gates & acceptance criteria

**H1 exit (installable PWA):**
- Airplane-mode phone: app opens, last chart renders, tarot draw + sigil +
  numerology + Resonarium fully functional, AI asks queue and flush on
  reconnect. Zero external requests (fonts self-hosted). Lighthouse PWA ≥ 95.
- `frontend/e2e/` mobile-viewport suite green in CI.

**H2 exit (store-ready shell):**
- Capacitor builds (APK + IPA) from `main` in CI; share-sheet PDF export;
  signed-token verification working with Ed25519 dual-issue live.
- Store-policy decision recorded per platform in this file.

**H3 exit (true counterpart):**
- `@astra/core` parity CI green across all engine vectors; chart cast,
  forecast, and full tarot suite complete on-device in airplane mode.
- Backend demoted to: AI relay + entitlement mint + optional sync. The
  privacy claim ("birth data never leaves the device") is now *structural*,
  not procedural.
